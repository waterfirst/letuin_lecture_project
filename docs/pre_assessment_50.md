# AI 바이브코딩 강의 사전평가 50문항

이 문서는 저장소의 강의 구조(`lectures/part1`~`part6`, `projects/project1`~`project3`, `docs/pre_test.md`)를 바탕으로 만든 강의 전 사전평가 문제집이다.
객관식과 주관식을 혼합했으며, 각 문항마다 정답과 간단한 해설을 포함했다.

## 구성

- 객관식 30문항
- 주관식 20문항
- 범위: AI/바이브코딩, GitHub, Python/Streamlit, Quarto, API, 데이터 분석, 배포, 보안, Telegram Bot, 디스플레이 공정

## 2. 주관식

### 31번
이 저장소에서 `lectures/`, `projects/`, `resources/` 폴더가 각각 어떤 역할을 하는지 간단히 설명하시오.

정답 예시: `lectures/`는 강의 슬라이드와 파트별 설명, `projects/`는 단계별 실습 프로젝트, `resources/`는 치트시트 같은 보조자료를 담는다.
해설: 저장소 구조를 이해하면 어떤 문서를 먼저 읽고 어떤 예제를 참고할지 빠르게 판단할 수 있다.

### 32번

정답 예시: AI의 출력 품질은 입력 지시의 구체성에 크게 좌우되기 때문이다. 역할, 목표, 출력 형식을 명확히 주면 수정 횟수를 줄일 수 있다.
해설: 바이브코딩은 코드를 직접 다 치는 대신 요구사항을 정확히 전달하는 역량이 핵심이다.

### 33번
`git commit`과 `git push`의 차이를 설명하시오.

정답 예시: `git commit`은 로컬 저장소에 변경 이력을 기록하는 것이고, `git push`는 그 이력을 원격 저장소로 올리는 것이다.
해설: 둘을 구분하지 못하면 버전 관리 흐름 전체가 흔들린다.

### 34번
Streamlit을 활용하면 비전공자나 초급자에게 어떤 장점이 있는지 설명하시오.

정답 예시: 복잡한 프론트엔드 없이 Python 코드만으로 웹 UI를 빠르게 만들 수 있어 시제품 제작 속도가 빠르다.
해설: 저장소의 여러 프로젝트가 Streamlit을 중심으로 구성된 이유이기도 하다.

### 35번

정답 예시: 필요한 패키지나 버전을 맞추지 못해 실행 오류가 발생할 수 있고, 팀원마다 환경이 달라 재현성이 떨어진다.
해설: 의존성 문서는 협업과 배포의 기본이다.

### 36번

정답 예시: API Key, 토큰, 비밀번호 같은 민감 정보를 실수로 GitHub에 올리는 것을 막기 위해서다.

해설: 공정 데이터처럼 변동성과 이상 상태가 중요한 데이터에서 특히 그렇다.

해설: 강의 후반의 모니터링, 경보, 알림 자동화와 직접 연결된다.

API를 앱에 연동할 때 "인증"이 필요한 이유를 설명하시오.

해설: 강의에서 결과물 유형에 따라 배포 플랫폼을 다르게 선택하도록 안내한다.
